简述

本文章仅供参考（都是自己瞎玩的），此外提到卡常，就不得不提到每次都把自己卡死的 RBtree 了。所以经常出现一道题全机房只有 RBtree TLE 1个点的情况（捂脸）。

浅谈

其实RBtree总是被卡是有原因的——他很喜欢一些奇怪的卡常方法或者语法，经过对他代码的观察和网络上的资料，我总结了一下几点注意事项：

  • 能用pair就不要用结构体，能用结构体就不要用类。

  • 即使scanf在格式化读入的时候很方便，但是不可能比关闭同步流的cin块，而且其他的c++读入是比c的读入要快很多的。

  • 关闭同步流 cpp ios::sync_with_stdio(false);cin.tie(0)

  • 关于快读 …\dots… 能不用就不用吧，因为卡读入的出题人是**。

  • 在要频繁调用一个数组的时候，可以先 memsetmemsetmemset 一遍，将其放入缓存中，增加速度。

  • 在循环变量前加入register（慎用）。

  • 尽可能养成优化代码的习惯，一个好的代码习惯也许可以加快运行速度。

  • define 比 const int 要更快一点点（其实区别不大）。

  • 可以将bool数组换成bitset，不仅支持更多功能，而且内存和时间都会减少。

  • 如果要用 rand 函数，可以用

代替，随机数更快而且更均匀。

最后，如果收集到更多关于卡常的知识的话，我会继续分享的。
